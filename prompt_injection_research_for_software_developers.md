# Securing agentic coding tools against prompt injection today

_This document was originally generated by Anthropic's Claude using Deep Research. Make of that what you will._

**Every major agentic coding tool - Claude Code, Cursor, Copilot, Devin, and others - has been successfully exploited via prompt injection in real-world research.** The "lethal trifecta," a term coined by [Simon Willison in June 2025](https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/), describes why: these tools simultaneously access private data, ingest untrusted content, and communicate externally - a combination that makes data exfiltration trivially achievable once an injection lands. The good news is that practical, deployable-today mitigations exist for each leg of the trifecta. This guide covers specific configurations, tools, and workflows that development teams can implement immediately to dramatically reduce risk, organized by the three legs of the trifecta plus process and tooling controls.

## The trifecta: why coding agents are uniquely dangerous

The lethal trifecta names the architectural reality that makes agentic coding tools fundamentally different from traditional dev tools. A coding agent that can read your `.env` files (private data), process a malicious GitHub issue (untrusted content), and execute `curl` (external communication) gives an attacker a complete kill chain with no further exploitation needed. Johann Rehberger's ["Month of AI Bugs"](https://embracethered.com/blog/posts/2025/wrapping-up-month-of-ai-bugs/) in August 2025 proved this systematically: he published one vulnerability per day across ChatGPT Codex, Claude Code, GitHub Copilot, Cursor, Devin, OpenHands, Google Jules, Amazon Q, AWS Kiro, Windsurf, and Cline. **Every single tool fell.**

The attacks are not theoretical. [CVE-2025-53773](https://embracethered.com/blog/posts/2025/github-copilot-remote-code-execution-via-prompt-injection/) demonstrated wormable remote code execution in GitHub Copilot, where a malicious README could modify VS Code settings to enable auto-approve mode, then execute arbitrary shell commands - and propagate itself through repositories. [Devin was exploited](https://embracethered.com/blog/posts/2025/devin-can-leak-your-secrets/) via zero-click data exfiltration through GitHub issues that caused it to `curl` secrets to attacker servers. Claude Code had API keys exfiltrated via DNS requests ([CVE-2025-55284](https://securityaffairs.com/188508/security/untrusted-repositories-turn-claude-code-into-an-attack-vector.html)) and project configuration files ([CVE-2026-21852](https://research.checkpoint.com/2026/rce-and-api-token-exfiltration-through-claude-code-project-files-cve-2025-59536/)).

The core defense strategy is simple: **break any one leg of the trifecta**. An agent that cannot reach the network cannot exfiltrate stolen secrets. An agent that never sees credentials has nothing valuable to steal. An agent that never processes untrusted input receives no malicious instructions. The mitigations below target each leg independently, so implementing even a subset provides meaningful protection.

## Restricting access to sensitive data

The first leg of the trifecta - private data access - is the most immediately actionable to restrict. Every tool has some mechanism for file exclusion, but their reliability varies dramatically.

**Claude Code** does not reliably honor `.claudeignore` files. As [reported by The Register in January 2026](https://www.theregister.com/2026/01/28/claude_code_ai_secrets_files/), Claude Code reads `.env` files even when `.claudeignore` entries should block access. The officially supported mechanism is **permission deny rules** in `settings.json` ([Claude Code permissions docs](https://code.claude.com/docs/en/permissions)):

```json
{
  "permissions": {
    "deny": [
      "Read(.env)", "Read(.env.*)", "Read(.envrc)",
      "Read(./secrets/**)", "Read(~/.ssh/**)", "Read(~/.aws/**)",
      "Read(~/.config/gcloud/**)", "Read(**/*.key)", "Read(**/*.pem)",
      "Read(.npmrc)", "Read(.pypirc)"
    ]
  }
}
```

These deny rules are evaluated first (before allow or ask rules) and cannot be overridden by the agent. Place them in `~/.claude/settings.json` for global protection as a user, and in `.claude/settings.json` at the project level for team enforcement. Enterprise teams can deploy **managed settings** via MDM profiles (Jamf, Kandji on macOS) or Group Policy on Windows, which take the highest precedence and cannot be overridden locally.

**Cursor** offers `.cursorignore` with `.gitignore` syntax, plus a separate `.cursorindexingignore` for controlling what gets indexed. However, files open in editor tabs may still be accessible to the AI despite ignore rules - the exclusion mechanism controls the AI knowledge base, not a hard security boundary. **GitHub Copilot** provides the strongest enterprise story: organization administrators can configure content exclusion patterns centrally ([GitHub Copilot policy docs](https://docs.github.com/copilot/managing-copilot/managing-github-copilot-in-your-organization/managing-github-copilot-features-in-your-organization/managing-policies-for-copilot-in-your-organization)), applying `.gitignore`-style rules across all organization members without relying on individual developer configuration. **Windsurf** uses `.codeiumignore`. **Aider** and **Amazon Q Developer** have no documented file exclusion mechanisms - with Aider, `.gitignore` affects repo mapping but is not a security boundary.

The files and directories that should **never** be exposed to any coding agent include: `.env*`, `*.key`, `*.pem`, `.ssh/`, `.aws/`, `.npmrc`, `.pypirc`, `.docker/config.json`, `.kube/config`, `.gnupg/`, `config/secrets/`, and `credentials.json`. But file exclusion alone is insufficient. The more robust approach is **credential injection at runtime**, where the agent never handles raw secrets at all. Tools like [**1Password CLI**](https://1password.com/solutions/agentic-ai) now offer agentic AI support with dynamic secret retrieval per task, and **HashiCorp Vault** can inject short-lived credentials via sidecar proxies. The pattern [Anthropic recommends](https://platform.claude.com/docs/en/agent-sdk/secure-deployment) for Claude Code in containers is a Unix socket proxy that sits outside the sandbox boundary, intercepts outbound API requests, and injects credentials - the agent sees only environment variable references, never actual keys.

## Limiting exposure to untrusted content

The second leg - untrusted content - is the hardest to eliminate because it's often the reason the agent is useful. Code comments, README files, GitHub issues, PR descriptions, package documentation, and configuration files like `CLAUDE.md` or `.cursorrules` are all proven injection vectors.

**CLAUDE.md files in cloned repositories are a documented prompt injection vector.** [Secure Code Warrior research](https://www.securecodewarrior.com/article/prompt-injection-and-the-security-risks-of-agentic-coding-tools) showed that files titled "Important! Read me!.md" buried in repositories would be read by agents hunting for context, executing embedded instructions. [Check Point Research](https://research.checkpoint.com/2026/rce-and-api-token-exfiltration-through-claude-code-project-files-cve-2025-59536/) demonstrated that `.claude/settings.json` files (which can contain hooks and MCP configurations) could achieve remote code execution before any trust dialog appeared (CVE-2025-59536, patched August 2025). The mitigation is straightforward: **always review CLAUDE.md and .claude/ directory contents before opening any cloned or forked repository with Claude Code**. The same applies to `.cursorrules` files (Cursor processes these without validation) and `.vscode/settings.json` (which Copilot can be tricked into modifying to enable auto-approve mode).

For MCP (Model Context Protocol) servers, the risks are severe. [Snyk's February 2026 "ToxicSkills" audit](https://snyk.io/blog/toxicskills-malicious-ai-agent-skills-clawhub/) found that **13.4% of all agent skills contain critical security issues** including malware and prompt injection. Never set `enableAllProjectMcpServers: true` in Claude Code settings. Only enable MCP servers you've personally reviewed and trust. Block risky servers proactively with `"disabledMcpjsonServers": ["filesystem"]` in settings. [Tim Kellogg's "MCP Colors" framework](https://timkellogg.me/blog/2025/11/03/colors) offers a useful mental model: classify tools as **red** (exposes agent to untrusted instructions) or **blue** (performs critical actions), and never allow both colors in a single session.

To minimize untrusted content exposure across tools:

- **Cursor**: Beware Auto-Run mode (the single biggest security risk - [academic research](https://arxiv.org/html/2509.22040v1) recently found prompt injection success rates of **55.6% to 93.3%** when Cursor is in Auto Mode). Enable MCP Tool Protection to require explicit approval for MCP tool invocations.
- **GitHub Copilot**: The coding agent is disabled by default on Business/Enterprise plans and must be explicitly enabled by an administrator - leave it off unless needed. When enabled, it creates draft PRs rather than committing directly ([Copilot coding agent access docs](https://docs.github.com/en/copilot/concepts/agents/coding-agent/access-management)).
- **Devin**: Avoid assigning Devin to work on issues from public repositories without reviewing issue content first. [Devin's unrestricted internet access](https://embracethered.com/blog/posts/2025/devin-can-leak-your-secrets/) means it can navigate to any URL an attacker specifies in an issue.
- **All tools**: Treat any content from public repositories, issues, or PRs as potentially hostile. Use Plan/read-only mode when analyzing unfamiliar codebases.

## Controlling external communication and exfiltration

The third leg - external communication - is the most effectively restricted through sandboxing and network controls.

**Claude Code's built-in sandbox** ([sandboxing docs](https://code.claude.com/docs/en/sandboxing), [Anthropic engineering post](https://www.anthropic.com/engineering/claude-code-sandboxing)) (released October 2025) provides OS-level isolation using bubblewrap on Linux and seatbelt profiles on macOS. It restricts filesystem access to the current working directory and routes all network traffic through a Unix domain socket proxy that enforces domain allowlists. Enable it in `settings.json`:

```json
{
  "sandbox": {
    "enabled": true,
    "network": {
      "allowedDomains": ["github.com", "*.npmjs.org", "api.anthropic.com"],
      "allowLocalBinding": true
    }
  }
}
```

For stronger isolation, **DevContainers** are Anthropic's recommended approach ([DevContainer docs](https://code.claude.com/docs/en/devcontainer)). A properly hardened DevContainer drops all Linux capabilities, blocks `sudo`, prevents SSH agent forwarding, uses a Docker socket proxy [Tecnativa](https://github.com/Tecnativa/docker-socket-proxy) for read-only Docker access, and implements iptables-based egress filtering.

[**Docker Sandboxes**](https://docs.docker.com/ai/sandboxes) offer the most turnkey solution, providing microVM-based isolation specifically for coding agents:

```bash
docker sandbox run code
```

Each sandbox gets its own Docker daemon, network isolation with domain allow/deny lists, and workspace syncing. This is the fastest path to meaningful isolation for teams that don't want to configure DevContainers from scratch.

For network restrictions specifically, block `curl`, `wget`, and `nc` in Claude Code's deny rules. In containers, implement iptables rules with a default-deny outbound policy and explicit allowlists for necessary domains (API endpoints, package registries, source control). Block internal metadata services to prevent SSRF. The [Anthropic-recommended pattern](https://platform.claude.com/docs/en/agent-sdk/secure-deployment) uses an Envoy proxy outside the sandbox boundary that validates requests, enforces domain allowlists, and injects credentials - so the agent communicates only through a controlled channel that logs everything.

Across tools, network restriction capabilities vary significantly. **OpenAI Codex** [blocks internet access by default](https://developers.openai.com/codex/cloud/internet-access/) during agent execution (allowing it only during setup for dependency installation). **[OpenHands](https://openhands.dev/)** runs in Docker containers with configurable network isolation. **Cursor, Aider, and Amazon Q Developer** provide no native network egress controls - external container or firewall-level restrictions are required.

## Workflow and process mitigations that reduce blast radius

Even with technical controls, workflow design is critical for containing damage when (not if) an injection succeeds.

**Human-in-the-loop approval** is necessary but insufficient alone. Claude Code's default "ask" mode requires approval for every risky action, but [CVE-2025-53773](https://embracethered.com/blog/posts/2025/github-copilot-remote-code-execution-via-prompt-injection/) showed that Copilot could be tricked into modifying its own configuration to bypass approvals. [Amazon Q's RCE exploit](https://www.theregister.com/2025/08/20/amazon_quietly_fixed_q_developer_flaws/) showed that `find -exec` and `ping` (used for DNS exfiltration) were classified as safe commands and executed without approval. The lesson: **allowlisting "safe" commands without a sandbox is fundamentally flawed** - commands like `find`, `grep`, and `git` all have parameters enabling code execution.

Claude Code offers distinct modes for managing risk. **Plan mode** is read-only - Claude analyzes and plans without making changes. Use it for code review, architecture analysis, and unfamiliar codebases. **Accept Edits mode** auto-approves file edits while still prompting for shell commands. **Normal mode** (default) prompts for everything. The `--dangerously-skip-permissions` flag should **only** be used inside ephemeral containers in CI/CD pipelines, never on local machines.

**Git branch isolation** is essential. Run all agent work on dedicated feature branches (`agent/<task-name>`), require PR review before merging, and never mount SSH keys into agent containers so `git push` is physically impossible from inside the sandbox. Review `git diff` output before every commit of agent-generated code.

**Pre-commit hooks** provide a critical safety net. For example, install [**gitleaks**](https://github.com/gitleaks/gitleaks) for secret scanning, [**Semgrep**](https://github.com/returntocorp/semgrep) for multi-language static analysis, and [**Bandit**](https://github.com/PyCQA/bandit) for Python security checks. These catch secrets or insecure patterns that agents introduce before they reach version control:

```yaml
repos:
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.22.1
    hooks:
      - id: gitleaks
  - repo: https://github.com/returntocorp/semgrep
    rev: v1.90.0
    hooks:
      - id: semgrep
        args: ['scan', '--config', 'auto']
```

In CI/CD pipelines, run SAST tools on all PRs from agent branches, verify dependencies against known-good registries (agents can hallucinate malicious package names), and generate SBOMs for supply chain visibility.

## Open-source guardrails and monitoring tools worth deploying

Several open-source tools specifically address agentic AI security and are production-ready today.

- [**Meta's LlamaFirewall**](https://ai.meta.com/research/publications/llamafirewall-an-open-source-guardrail-system-for-building-secure-ai-agents/) combines three components: PromptGuard 2 for jailbreak detection, Agent Alignment Checks for chain-of-thought auditing that catches prompt injection and goal misalignment, and CodeShield for online static analysis preventing insecure code generation. It's the most comprehensive open-source guardrails framework available.
- [**NVIDIA NeMo Guardrails**](https://github.com/NVIDIA-NeMo/Guardrails) provides programmable input, dialog, retrieval, execution, and output rails using the Colang modeling language, with LangChain and LlamaIndex integration.
- [**Guardrails AI**](https://github.com/guardrails-ai/guardrails) offers a Python framework with a community-driven validator hub for PII detection, toxic language filtering, and regex matching.

For sandboxing infrastructure:
- The [**Kubernetes agent-sandbox**](https://github.com/kubernetes-sigs/agent-sandbox) controller (from kubernetes-sigs) provides isolated agent runtimes with gVisor or Kata Containers support for enterprise Kubernetes deployments.
- [**E2B**](https://e2b.dev/) offers open-source cloud sandboxes with Python and JavaScript SDKs for sandboxed code execution.
- [**Coder Agent Boundaries**](https://coder.com/blog/launch-dec-2025-agent-boundaries) provides an enterprise agent-aware firewall that intercepts every request between AI agents and the network, enforcing per-agent policies.

For monitoring:
- Claude Code supports [**OpenTelemetry**](https://opentelemetry.io/) for structured observability (prompts redacted by default).
- [**Langfuse**](https://langfuse.com/) (open-source) provides LLM trace logging and prompt/response monitoring.
- Pre-commit tools like [**gitleaks**](https://github.com/gitleaks/gitleaks), [**TruffleHog**](https://github.com/trufflesecurity/trufflehog), and **detect-secrets** catch secrets before they reach version control.

## Conclusion: defense in depth, implemented in layers

The most important insight from this research is that **no single mitigation is sufficient**, but the combination of even basic controls across all three trifecta legs provides substantial protection. An agent running in a DevContainer with network egress filtering, deny rules blocking credential file access, and pre-commit hooks catching secrets represents a dramatically smaller attack surface than an unconfigured agent running on a developer's host machine with full network access.

The fundamental architectural insight from the research community - [Meta's "Rule of Two"](https://ai.meta.com/blog/practical-ai-agent-security/), [Willison's trifecta-breaking strategy](https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/), and [Beurer-Kellner's six design patterns](https://langfuse.com/) - converges on one principle: **once an agent has ingested untrusted input, it must be impossible for that input to trigger consequential actions**. Until LLMs can reliably distinguish instructions from data (which no current approach achieves), the only reliable defenses are architectural ones that prevent the dangerous combination of capabilities from existing in a single session.
